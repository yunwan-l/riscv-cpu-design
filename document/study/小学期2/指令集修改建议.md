## 不必要指令分析

首先明确题目B的三个层次对应的指令需求：

| 层次 | 要求 | 必要指令范围 |
|------|------|------------|
| **基础** | "支持算术、逻辑、访存、跳转等基本指令类型，能运行简单测试程序" | RV32I 极小核心子集 |
| **进阶** | 引入流水线 + 集成 I/O + 性能量化 | RV32I 完整子集 |
| **拓展** | "支持浮点运算或乘除法扩展指令" | M 扩展等（**可选项**） |

---

### 一、完全不必要 — M 扩展（8条）（我觉得不可以删，要保留算数）

| 指令 | 理由 |
|------|------|
| **mul, mulh, mulhsu, mulhu** | |
| **div, divu, rem, remu** | |

**理由：** 课程要求写得非常清楚——乘除法扩展是"拓展层次"的**可选方向之一**：

> 拓展层次——高性能处理器优化设计：...从以下方向中选择至少一项深入实现：...Cache替换策略优化与命中率分析；**支持浮点运算或乘除法扩展指令**；...面向AI/矩阵计算的自定义加速指令设计...

这意味着乘除法不是必选项。项目当前选了 Cache 替换策略作为拓展方向，M 扩展对验收评分**没有贡献**，反而会导致：

- **译码器变复杂**：OP 操作码下需要额外判 `funct7==0000001`，增加关键路径
- **EX 级需要乘除法单元**：乘法器面积大（在 Artix-7 上一个 32 位乘法器约消耗 4 个 DSP slice），除法器更占用资源
- **控制信号膨胀**：`ctrl_t` 多了 `use_multdiv` + `multdiv_op` 两个字段

**建议：** 直接删掉 M 扩展，`rvp_configs.yaml` 中所有配置的 `RV32M` 改为 0。

---

### 二、基本不必要 — RV32I 中可省略的指令（16条）

#### 2.1 字节/半字访存（6条）（我觉得可以删）

| 指令 | 理由 |
|------|------|
| **lb, lbu** | 加载 8 位（有符号/无符号） |
| **lh, lhu** | 加载 16 位（有符号/无符号） |
| **sb, sh** | 存储 8 位/16 位 |

**理由：** 题目B的运行环境是 NEXYS4 开发板，项目使用 32 位 BRAM 作为内存。所有外设（UART/GPIO/Timer）的寄存器也都是 32 位对齐的——代码中 `bus_be_o` 基本只用 `4'b1111`。**没有 8 位或 16 位访存的真实需求**。

删除后对项目的影响：
- **MEM 级大幅简化**：去掉 150 行字节使能生成 + Store 对齐 + Load 符号扩展的 `unique case` 分支
- **译码器化简**：LOAD/STORE 分支各少 2~3 个 funct3 匹配
- **`ctrl_t` 结构体**：`mem_unsigned` 字段可以删除，`mem_size` 枚举只剩 `SIZE_W` 一个值→可以整个删掉

实测：项目 42 个测试用例中，`lb.S`/`lbu.S`/`lh.S`/`lhu.S`/`sb.S`/`sh.S` 这 6 个测试都用不到。

#### 2.2 无条件比较置位（4条）（我觉得不用删，不太影响性能）

| 指令 | 理由 |
|------|------|
| **slt, sltu** | 有符号/无符号小于比较 |
| **slti, sltiu** | 立即数有符号/无符号小于比较 |

**理由：** 这些指令把比较结果存回寄存器，可以用**分支指令+条件赋值**替代：

```asm
# slt x5, x6, x7  等价于：
  blt x6, x7, 1f
  addi x5, x0, 0      # 不成立 → x5=0
  j    2f
1: addi x5, x0, 1      # 成立 → x5=1
2:
```

代价是 3~4 条指令，但在"运行简单测试程序"的场景下完全可以接受。外设驱动和测试程序以功能正确性为主，不需要这类指令来优化。

ALU 中的 `ALU_SLT`/`ALU_SLTU` 操作码可以删掉（但比较逻辑 `is_less_signed`/`is_less_unsigned` 保留，分支指令还要用）。

#### 2.3 多余的分支条件（2条）（我觉得不删）

| 指令 | 理由 |
|------|------|
| **bltu, bgeu** | 无符号小于/大于等于分支 |

**理由：** 保留 beq/bne/blt/bge 四条分支已经足够。无符号比较分支在基本测试程序中几乎用不到。对于本项目的内存地址比较（地址都是正数），有符号比较 `blt` 已经能正确处理。

#### 2.4 异或指令（2条）（我觉得不删）

| 指令 | 理由 |
|------|------|
| **xor, xori** | 按位异或 |

**理由：** 异或在基本程序中的典型用途（清零寄存器、位翻转）可以用 `addi rd, x0, 0`（清零）或 `xori rd, rs1, -1`（取反）等替代。但 `xori` 本身依赖立即数，如果已经有了 `addi`/`andi`/`ori`，保留 `xori` 的成本极低（译码器里多一行），反而有必要性。所以只建议删 `xor`（R型），`xori` 保留。

#### 2.5 算术右移（2条）（我觉得删，然后替代）

| 指令 | 理由 |
|------|------|
| **sra, srai** | 算术右移 |

**理由：** 算术右移主要用于有符号数除以 2 的幂次。在基本测试程序中，除法不常用，且可以用逻辑右移+符号位修复替代。保留 `sll/slli`（左移，地址对齐计算必需）和 `srl/srli`（逻辑右移，位提取）就够了。删除后 ALU 少一个 `$signed >>>` 路径。

---

### 三、有争议但建议保留的（9条）

| 指令 | 保留理由 |
|------|---------|
| **and, or** | 寄存器间与/或，外设寄存器位字段操作常用 |
| **andi, ori** | 立即数与/或，对外设 MMIO 地址构建（`0x10000000`）和位掩码是刚需 |
| **sll, slli** | 左移，字节地址→字索引转换（×4）必须 |
| **srl, srli** | 逻辑右移，提取位域，gcc 编译的代码中会出现 |
| **blt, bge** | 有符号小于/大于等于分支，替代 slt 类指令需要 |
| **auipc** | PC 相对寻址，配合 jalr 做长跳转。虽然可用 lui+jalr 等效替代，但标准 RISC-V ABI 依赖它 |

---

### 四、必不可少的核心指令（12条）

这些是"运行简单测试程序 + 驱动外设"的绝对最低配置：

```
add, addi, sub        — 算术运算（地址计算、计数器）
lw, sw                — 32位内存读写（包括外设MMIO）
beq, bne              — 相等/不等分支（任何条件逻辑的基础）
jal, jalr             — 函数调用/返回
lui                   — 构建大立即数（外设地址如 0x10000000）
andi, ori             — 位掩码（外设寄存器置位/清零）
```

---

### 总结

| 类别 | 数量 | 指令 |
|------|------|------|
| **必须保留** | ~21条 | RV32I 核心子集：add/addi/sub/and/andi/or/ori/xori/sll/slli/srl/srli/slt/slti/lw/sw/lui/auipc/beq/bne/blt/bge/jal/jalr |
| **建议删除** | M扩展8条 + RV32I 16条 ≈ 24条 | M扩展全部 + lb/lbu/lh/lhu/sb/sh + sltu/sltiu/bltu/bgeu + sra/srai + xor |

删除后的直接收益：
- **译码器** `rvp_decoder.sv` 从 ~275 行缩减到约 120 行
- **ALU** `rvp_alu.sv` 去掉 M 扩展接口和 4 种操作码
- **MEM 级** `rvp_mem_stage.sv` 去掉全部字节/半字对齐逻辑
- **`ctrl_t`** 删除 `use_multdiv`、`multdiv_op`、`mem_unsigned`、`mem_size` 等字段
- **`rvp_pkg.sv`** 删除 `multdiv_op_e` 枚举
