# 优化前后对比分析

## 1. 优化方案概述

### 1.1 优化方法：指令调度消除 Load-Use 停顿

本次优化采用**编译器级指令调度**，在汇编层面重排独立指令，填充 Load-Use 停顿周期。该方案无需修改硬件，仅通过软件层面的指令重排实现性能提升。

**优化原理**：

Load 指令在 MEM 阶段（第 4 级）才能读到数据，若下一条指令在 EX 阶段（第 3 级）需要该数据，前递无法跨越 MEM→EX 的时序边界，需插入 1 个停顿周期。通过在 Load 和依赖指令之间插入一条独立指令（无数据依赖），可让 Load 数据在下一周期从前递通路就绪，消除停顿。

**优化前流水线**：

```
时钟周期:    1     2     3     4     5     6     7
lw 指令:    [IF]  [ID]  [EX]  [MEM] [WB]
                              ↑ 数据可用
依赖指令:          [IF]  [ID]  [停顿] [EX]  [MEM] [WB]
                               ↑ stall=1, 等待前递
损失: 1 个时钟周期
```

**优化后流水线**：

```
时钟周期:    1     2     3     4     5     6     7
lw 指令:    [IF]  [ID]  [EX]  [MEM] [WB]
                              ↑ 数据可用
独立指令:          [IF]  [ID]  [EX]  [MEM] [WB]
依赖指令:               [IF]  [ID]  [EX]  [MEM] [WB]
                                     ↑ t1 从 MEM/WB 前递，无停顿
损失: 0 个时钟周期
```

### 1.2 各程序优化点

**矩阵乘法优化**：

内层循环中 `lw t1, 0(t5)` → `mul t4, t0, t1` 存在 Load-Use 冒险。将 `addi a2, a2, 1`（k++，与 t1 无数据依赖）从 `mul/add` 之后移至 `lw t1` 与 `mul t4` 之间，填充停顿槽。

```assembly
# 优化前（产生 Load-Use 停顿）：
lw t1, 0(t5)       # Load B[k][j]
mul t4, t0, t1     # 依赖 t1 → 停顿 1 周期
add t2, t2, t4
addi a2, a2, 1     # k++

# 优化后（消除停顿）：
lw t1, 0(t5)       # Load B[k][j]
addi a2, a2, 1     # 独立指令，填充停顿槽
mul t4, t0, t1     # t1 已就绪（MEM/WB 前递），无停顿
add t2, t2, t4
```

**冒泡排序优化**：

内层循环中 `lw t1, 0(t4)` → `ble t0, t1` 存在 Load-Use 冒险。将 `slli t4, a1, 2`（计算 j*4，交换代码用，与 t1 无数据依赖）从 `ble` 之后移至 `lw t1` 与 `ble` 之间，填充停顿槽。

```assembly
# 优化前（产生 Load-Use 停顿）：
lw t1, 0(t4)        # Load arr[j+1]
ble t0, t1, no_swap # 依赖 t1 → 停顿 1 周期
slli t4, a1, 2      # j*4（交换代码用）

# 优化后（消除停顿）：
lw t1, 0(t4)        # Load arr[j+1]
slli t4, a1, 2      # 独立指令，填充停顿槽
ble t0, t1, no_swap # t1 已就绪，无停顿
# 若 ble 跳转（无需交换），t4 结果不使用，无副作用
# 若 ble 不跳转（需要交换），t4 已就绪，sw 直接使用
```

**斐波那契**：无 Load-Use 停顿（stall=0），无需优化，原版与优化版相同。

### 3.4 冒泡排序指令数变化分析

冒泡排序优化版指令数为 497（原版 482），增加 15 条。原因：优化版将 `slli t4, a1, 2` 从交换代码块移至 `ble` 之前，该指令在每次内层循环迭代中都会执行（无论是否交换）。在无需交换的迭代中，原版跳过该指令，优化版多执行 1 条。多出的 15 条指令对应 15 次无交换迭代。

尽管指令数增加，但由于消除了 28 次停顿周期（每次停顿浪费 1 周期），净效果仍为周期数减少 13（28 - 15 = 13），CPI 降低 5.24%。

## 5. 结论

### 5.1 优化成效

1. **Load-Use 停顿完全消除**：矩阵乘法和冒泡排序的 Load-Use 停顿从 27/28 次降至 0 次，验证了指令调度优化的正确性
2. **CPI 显著降低**：矩阵乘法 CPI 降低 3.70%（1.0846→1.0445），冒泡排序 CPI 降低 5.24%（1.1826→1.1207）
3. **吞吐量提升**：在 12.5 MHz 时钟频率下（实际 SoC 时钟），平均 MIPS 从 10.98 提升至 11.33（+3.10%）
4. **数据正确性验证**：全部 6 个 benchmark 数据验证 6/6 PASS，证明指令调度未改变程序语义

### 5.2 优化局限性

1. **仅消除数据冒险**：指令调度只能消除 Load-Use 停顿（数据冒险），对分支冲刷（控制冒险）无影响
2. **依赖独立指令可用性**：若 Load 后无法找到合适的独立指令填充停顿槽，则无法优化
3. **斐波那契无优化空间**：该程序无 Load-Use 停顿，指令调度无法改善

### 5.3 后续优化方向

指令调度优化已将数据冒险惩罚降至 0，当前剩余 CPI 损失主要来自**控制冒险（分支冲刷）**，占额外 CPI 的 66.2%。要进一步提升性能，需从硬件层面入手：

| 优化方案 | 目标 | 预期 CPI 改善 | 实现复杂度 |
|----------|------|-------------|-----------|
| 分支判定提前至 ID | 消除冲刷 | 大 | 中 |
| 2-bit 饱和预测器 | 减少冲刷 | 大 | 中高 |
| 提频至 100 MHz | 提高频率 | 无（CPI不变） | 低 |
| 组合：提频+分支提前 | 最大化 | 大 | 中 |

> 推荐优先实施"提频至 100 MHz + 分支判定提前至 ID"，预期吞吐量提升 2.17×。

### 5.4 板载验证状态

板载验证已于 2026-07-10 全部通过，8 项测试（含 ALU/分支/存储器/乘除法 UART 验证）均 PASS。实际 SoC 运行时钟为 12.5 MHz，时序满足（WNS +6.247ns，基于25MHz综合结果），I-Cache 已实现（256 字节直接映射，异步读，对 CPI 无影响）。
