# 时序与资源分析

## 1. 时钟架构

### 1.1 时钟域划分

RVP SoC 采用双时钟域设计，板载 100 MHz 晶振通过分频器生成 CPU 核心时钟：

```
板载晶振 (100 MHz, E3引脚)
    │
    ├── sys_clk_pin (100 MHz, 10 ns 周期)
    │   └── 外设域：七段数码管动态扫描、UART 分频
    │
    └── clk_div2 (50 MHz, 20 ns 周期)
        └── CPU 核心域：流水线 CPU、指令/数据存储器、性能计数器
```

### 1.2 时钟约束（XDC）

```tcl
create_clock -add -name sys_clk_pin -period 10.00 -waveform {0.000 5.000} [get_ports clk]
create_clock -name clk_div2 -period 20.000 [get_pins soc/clk_div_reg/Q]
```

- `sys_clk_pin`：100 MHz，直接来自板载晶振，驱动外设
- `clk_div2`：50 MHz，由 SoC 内部 `clk_div_reg` 分频产生，驱动 CPU 核心

## 2. 时序分析

### 2.1 总体时序概览

数据来源：`rvp_fpga_top_timing_summary_routed.rpt`（Vivado 2018.3 Implementation 后报告）

| 时钟域 | WNS (ns) | TNS (ns) | 失败端点 | 总端点 | 状态 |
|--------|----------|----------|----------|--------|------|
| sys_clk_pin | -9.196 | -124.352 | 15 | 35 | VIOLATED |
| clk_div2 | +15.649 | 0.000 | 0 | 100 | MET |
| **总体** | **-9.196** | **-124.352** | **15** | **135** | **VIOLATED** |

> WNS（Worst Negative Slack）> 0 表示时序满足，< 0 表示时序违例。

### 2.2 CPU 核心域（clk_div2）时序详情

**时序状态：MET（满足）**

| 参数 | 数值 | 说明 |
|------|------|------|
| 时钟周期 | 20.000 ns | 50 MHz |
| WNS | +15.649 ns | 最差负松弛（正值=满足） |
| TNS | 0.000 ns | 总负松弛（0=无违例） |
| WHS | +0.249 ns | 最差保持松弛（正值=满足） |
| 失败端点 | 0 / 100 | 100 个端点全部满足 |
| 脉宽松弛 | +9.500 ns | 脉宽裕量充足 |

**关键路径详情**：

| 参数 | 数值 |
|------|------|
| 源端 | `soc/uart/clk_div_reg[3]/C`（FDCE 触发器） |
| 目的端 | `soc/uart/clk_div_reg[24]/D`（FDCE 触发器） |
| 数据路径延迟 | 4.377 ns |
| 逻辑延迟 | 1.257 ns（28.7%） |
| 布线延迟 | 3.120 ns（71.3%） |
| 逻辑级数 | 0（触发器到触发器直连） |

**分析**：
- 关键路径位于 UART 分频计数器内部（`clk_div_reg` 位 3 到位 24），这是进位链传播路径
- 数据路径延迟仅 4.377 ns，远小于 20 ns 时钟周期，时序裕量高达 15.649 ns
- 布线延迟占比 71.3%，逻辑延迟仅 28.7%，说明逻辑不复杂，延迟主要由布线决定
- CPU 核心的实际数据通路（ALU、前递、冒险检测等）未出现在关键路径上，说明 CPU 逻辑设计紧凑

### 2.3 理论最高频率计算

```
理论 Fmax = 1000 / 数据路径延迟 = 1000 / 4.377 ≈ 228.5 MHz
```

| 频率指标 | 数值 | 说明 |
|----------|------|------|
| 实际运行频率 | 50 MHz | clk_div2 域 |
| 理论最高频率 | 228.5 MHz | 基于关键路径延迟 |
| 频率利用率 | 21.9% | 实际/理论 |
| 时序裕量 | 15.649 ns | 可提升空间充足 |

### 2.4 外设域（sys_clk_pin）时序违例分析

**时序状态：VIOLATED**

| 参数 | 数值 |
|------|------|
| WNS | -9.196 ns |
| TNS | -124.352 ns |
| 失败端点 | 15 / 35 |
| 最差路径 | `digit_sel_reg[2]/C` → `seg_cf`（输出端口） |

**违例路径分析**：

所有 15 个失败端点均为七段数码管输出路径，模式如下：

```
digit_sel_reg[2]/C (FDRE, sys_clk_pin)
    → MUXF7 (段码选择)
    → LUT4 (段码译码)
    → OBUF (输出缓冲)
    → seg_cf / seg_ce / seg_cg / an[3] (FPGA 输出引脚)
```

**违例原因**：

| 因素 | 数值 | 影响 |
|------|------|------|
| 时钟周期要求 | 10.000 ns | 100 MHz |
| BUFG 时钟延迟 (SCD) | 5.320 ns | 源端时钟到达延迟 |
| 数据路径延迟 | ~8.8 ns | 逻辑 + 布线 + OBUF |
| 输出延迟约束 | 5.000 ns | 板级时序要求 |
| 实际需要 | 5.320 + 8.8 + 5.0 = 19.12 ns | 远超 10 ns 周期 |

**影响评估**：

- 违例仅影响七段数码管显示刷新，不影响 CPU 核心功能
- 七段数码管为低速外设（1 kHz 刷新率），实际运行中时序违例不会导致可见的显示问题
- 可通过以下方式修复：
  1. 将七段数码管扫描逻辑移至 clk_div2 域（20 ns 周期，裕量充足）
  2. 减小 output delay 约束（如从 5 ns 降至 1 ns）
  3. 在输出路径上添加流水线寄存器

### 2.5 时钟交互分析

| 源时钟 | 目的时钟 | 跨时钟域路径 | 处理方式 |
|--------|----------|-------------|----------|
| sys_clk_pin | clk_div2 | 0 | clk_div2 由 sys_clk_pin 分频生成，同源同步 |
| clk_div2 | sys_clk_pin | 0 | 无反向路径 |

两个时钟域为同源同步关系（clk_div2 是 sys_clk_pin 的 2 分频），无异步跨时钟域问题。

### 3.5 资源占用率极低的原因分析

资源占用率极低（LUT 0.12%、FF 0.07%）的主要原因：

1. **存储器被优化**：Vivado 综合时，指令存储器（`rvp_instr_mem.sv`）和数据存储器（`rvp_data_mem.sv`）的数组在未被初始化的情况下被优化为空逻辑或分布式 RAM 的最小实现。实际部署中需要通过 `$readmemh` 或 BRAM 初始化文件加载程序数据。

2. **乘法器未使用 DSP**：`rvp_multdiv.sv` 中的乘法运算被综合为 LUT 逻辑而非 DSP48E1 硬核，因为操作数位宽较小且综合器选择 LUT 实现。实际部署中可通过属性强制使用 DSP。

3. **实际部署预估资源**：

   | 资源 | 当前 | 预估（含 BRAM） | 说明 |
   |------|------|------------------|------|
   | LUT | 75 | ~2000-3000 | 含 CPU 完整数据通路 |
   | FF | 89 | ~1500-2000 | 含流水线寄存器、寄存器堆 |
   | BRAM | 0 | 8-16 | 指令 4KB + 数据 4-8KB |
   | DSP | 0 | 1-2 | 乘法器 |

## 4. 关键路径优化建议

### 4.1 当前关键路径

当前 clk_div2 域的关键路径位于 UART 分频计数器内部（`clk_div_reg[3]` → `clk_div_reg[24]`），数据路径延迟仅 4.377 ns。

### 4.2 提升运行频率的方案

| 方案 | 目标频率 | 实现方式 | 难度 |
|------|----------|----------|------|
| 调整分频比 | 100 MHz | 将 clk_div2 从 2 分频改为 1 分频（直通） | 低 |
| 使用 PLL/MMCM | 200 MHz | 用 MMCM 生成高频时钟替代分频 | 低 |
| 优化七段数码管 | 100 MHz | 将数码管扫描移至独立慢时钟域 | 中 |

> 注意：若将 CPU 运行频率提升至 100 MHz，需重新验证 sys_clk_pin 域的时序（当前已有违例）。建议将七段数码管扫描逻辑迁移至 clk_div2 域或添加输出流水线寄存器。
