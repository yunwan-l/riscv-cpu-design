# 时序与资源分析

## 1. 时钟架构

### 1.1 时钟域划分

RVP SoC 采用双时钟域设计，板载 100 MHz 晶振通过 8 分频生成 12.5 MHz SoC 核心时钟：

```
板载晶振 (100 MHz, E3引脚)
    │
    ├── sys_clk_pin (100 MHz, 10 ns 周期)
    │   └── 外设域：七段数码管动态扫描
    │
    └── clk_soc (12.5 MHz, 80 ns 周期, 8 分频)
        └── CPU 核心域：流水线 CPU、I-Cache、指令/数据存储器、性能计数器
```

8 分频使用 3 位计数器实现，取最高位 `clk_cnt[2]` 作为 SoC 时钟，周期为 8 × 10ns = 80ns = 12.5MHz，占空比 50%。

### 1.2 时钟约束（XDC）

```tcl
# 板载晶振 100MHz
create_clock -add -name sys_clk_pin -period 10.000 -waveform {0 5} [get_ports { clk }];

# 分频时钟约束：clk_soc = clk / 8 = 12.5MHz (80ns period)
create_generated_clock -name clk_soc -source [get_ports { clk }] -divide_by 8 \
    [get_pins -hier -filter {NAME =~ *clk_cnt_reg[2]/Q}]

# 跨时钟域约束：SoC(12.5MHz) ↔ 数码管(100MHz)，设 false path
set_false_path -from [get_clocks clk_soc] -to [get_clocks sys_clk_pin]
set_false_path -from [get_clocks sys_clk_pin] -to [get_clocks clk_soc]

# 异步复位 false path
set_false_path -from [get_ports { rst_n }] -to [all_registers]

# 七段数码管为慢速显示设备，设 false path 避免时序分析
set_false_path -to [get_ports { seg_ca seg_cb seg_cc seg_cd seg_ce seg_cf seg_cg an[*] }]
```

- `sys_clk_pin`：100 MHz，直接来自板载晶振，驱动七段数码管扫描
- `clk_soc`：12.5 MHz，由 SoC 内部 `clk_cnt_reg[2]` 分频产生（8 分频），驱动 CPU 核心

> **false_path 约束说明**：跨时钟域路径（`clk_soc` ↔ `sys_clk_pin`）已添加双向 false_path 约束，因为 `pc_dbg` 等调试信号为慢变信号，无需满足同步时序要求。七段数码管输出（段码和位选）也设为 false_path，因为刷新率仅 1 kHz，人眼无法分辨纳秒级延迟。异步复位信号 `rst_n` 同样设为 false_path。

## 2. 时序分析

### 2.1 总体时序概览

数据来源：Vivado Implementation 后时序报告

| 时钟域 | WNS (ns) | TNS (ns) | 失败端点 | 总端点 | 状态 |
|--------|----------|----------|----------|--------|------|
| sys_clk_pin | — | 0.000 | 0 | — | MET |
| clk_soc | +6.247 | 0.000 | 0 | — | MET |
| **总体** | **+6.247** | **0.000** | **0** | — | **MET** |

> WNS（Worst Negative Slack）> 0 表示时序满足。所有时钟域时序均满足，无违例端点。

### 2.2 CPU 核心域（clk_soc）时序详情

**时序状态：MET（满足）**

| 参数 | 数值 | 说明 |
|------|------|------|
| 时钟周期 | 80.000 ns | 12.5 MHz（8 分频） |
| WNS | +6.247 ns | 最差负松弛（基于25MHz综合结果，正值=满足） |
| TNS | 0.000 ns | 总负松弛（0=无违例） |
| 失败端点 | 0 | 全部端点满足 |
| 状态 | MET | 时序通过 |

**分析**：
- 时钟周期 80 ns（当前），WNS 为 +6.247 ns（基于25MHz综合结果，即40ns周期约束），关键路径延迟为 40 - 6.247 = 33.753 ns
- 12.5 MHz 运行频率下时序裕量充足，关键路径延迟远小于时钟周期
- CPU 核心数据通路（ALU、前递、冒险检测、I-Cache 等）均在时序约束内

### 2.3 理论最高频率计算

```
关键路径延迟 = 时钟周期 - WNS = 40.000 - 6.247 = 33.753 ns（基于25MHz综合结果）
理论 Fmax = 1000 / 33.753 ≈ 29.63 MHz
```

| 频率指标 | 数值 | 说明 |
|----------|------|------|
| 实际运行频率 | 12.5 MHz | clk_soc 域（8 分频） |
| 理论最高频率 | 29.63 MHz | 基于关键路径延迟 33.753 ns（25MHz综合结果） |
| 频率利用率 | 42.2% | 实际/理论（12.5 / 29.63） |
| 时序裕量 | 6.247 ns | 可提升空间（基于25MHz综合结果） |

> **说明**：当前 8 分频方案将 SoC 时钟设为 12.5 MHz，理论最高频率为 29.63 MHz（基于25MHz综合结果），频率利用率 42.2%，时序裕量充裕。若需进一步提升频率，需优化关键路径或减少分频比（如改为 4 分频得到 25 MHz 或 2 分频得到 50 MHz），但需重新验证时序。

### 2.4 外设域（sys_clk_pin）时序分析

**时序状态：MET（满足）**

此前版本中，七段数码管输出路径（`digit_sel_reg` → 段码译码 → OBUF → 输出引脚）在 100 MHz 域存在时序违例（WNS = -9.196 ns）。当前版本通过 XDC 中添加 false_path 约束解决了此问题：

```tcl
set_false_path -to [get_ports { seg_ca seg_cb seg_cc seg_cd seg_ce seg_cf seg_cg an[*] }]
```

**处理依据**：
- 七段数码管为低速外设（1 kHz 刷新率），输出信号每 1 ms 才变化一次
- 人眼无法分辨纳秒级延迟，时序违例不影响显示效果
- false_path 约束使 Vivado 不再对这些路径做时序分析，消除违例

### 2.5 时钟交互分析

| 源时钟 | 目的时钟 | 跨时钟域路径 | 处理方式 |
|--------|----------|-------------|----------|
| sys_clk_pin | clk_soc | pc_dbg 等调试信号 | false_path（慢变信号，无需同步） |
| clk_soc | sys_clk_pin | 无反向数据路径 | false_path（双向已约束） |

两个时钟域为同源同步关系（clk_soc 是 sys_clk_pin 的 8 分频），但通过 false_path 约束避免了不必要的跨域时序分析。对于未来需要可靠跨域传输的信号，建议使用同步器（两级触发器）或异步 FIFO。

### 2.6 板级验证

在 Nexys4 DDR（Artix-7 XC7A100T）开发板上完成全系统验证：

| 验证项目 | 结果 | 说明 |
|----------|------|------|
| 全部 8 项功能测试 | 通过 | 涵盖算术运算、存储访问、分支跳转、前递链等 |
| I-Cache 缓存路径 | 通过 | 测试 3（前递链测试）通过缓存代码路径正确执行 |
| 七段数码管显示 | 正常 | PC 值实时显示，刷新稳定 |
| UART 输出 | 正常 | 12.5 MHz 时钟下波特率分频正确 |
| 性能计数器 | 正确 | I-Cache 命中/缺失计数器可通过总线读取 |

> 时序满足且板级验证全部通过，确认 12.5 MHz 运行频率下设计功能正确。

## 3. FPGA 资源利用率

### 3.1 实现后资源占用（Implementation 阶段）

数据来源：Vivado Implementation 后资源利用报告

| 资源类型 | 占用 | 可用 | 利用率 |
|----------|------|------|--------|
| Slice LUTs | 4,998 | 63,400 | 7.88% |
| Slice Registers (FF) | 5,477 | 126,800 | 4.32% |
| Block RAM | 1 | 135 | 0.74% |
| DSP48E1 | 12 | 240 | 5.00% |
| Slices | — | 15,850 | — |
| Bonded IOB | 34 | 210 | 16.19% |
| BUFG | 2 | 32 | 6.25% |

### 3.2 资源占用分析

与早期版本（存储器被优化、LUT 仅 75、FF 仅 89、BRAM 为 0、DSP 为 0）相比，当前实现为完整功能版本：

| 资源 | 早期版本（空设计） | 当前版本（完整实现） | 说明 |
|------|-------------------|---------------------|------|
| LUT | 75 (0.12%) | 4,998 (7.88%) | 含 CPU 完整数据通路、I-Cache、总线、外设 |
| FF | 89 (0.07%) | 5,477 (4.32%) | 含流水线寄存器、寄存器堆、I-Cache Tag/Data 阵列 |
| BRAM | 0 (0.00%) | 1 (0.74%) | 指令存储器（8KB）推断为 BRAM |
| DSP | 0 (0.00%) | 12 (5.00%) | 乘法器使用 DSP48E1 硬核实现 |

**资源使用说明**：

1. **BRAM（1 块）**：指令存储器（`rvp_instr_mem.sv`，2048 字 = 8KB）通过 `$readmemh` 初始化，被 Vivado 推断为单端口 BRAM。数据存储器（`rvp_data_mem.sv`）因支持异步读 + 同步写 + 子字访问，被推断为分布式 RAM（LUT 实现），不占用 BRAM 资源

2. **DSP（12 块）**：乘法器运算（`rvp_multdiv.sv`）使用 DSP48E1 硬核实现。12 个 DSP 用于 32 位乘法运算的流水化实现，相比 LUM 实现具有更低的延迟和更高的效率

3. **LUT/FF 分布**：
   - CPU 流水线数据通路（ALU、前递逻辑、冒险检测）：主要 LUT/FF 消耗
   - I-Cache（64 行 × Valid + Tag + Data）：约 64 × (1 + 24 + 32) = 3,648 位存储，使用寄存器/分布式 RAM 实现
   - 外设（UART、GPIO、Timer、性能计数器）：中等 LUT/FF 消耗
   - 七段数码管扫描逻辑：少量 LUT/FF

4. **整体利用率**：LUT 7.88%、FF 4.32%，资源占用适中，Artix-7 XC7A100T 芯片有充足余量用于功能扩展

## 4. 关键路径优化建议

### 4.1 当前时序状态

当前 clk_soc 域 WNS = +6.247 ns（基于25MHz综合结果），12.5 MHz 运行频率下时序满足。关键路径延迟约 33.753 ns，理论最高频率 29.63 MHz（基于25MHz综合结果）。

### 4.2 提升运行频率的方案

| 方案 | 目标频率 | 实现方式 | 难度 | 风险 |
|------|----------|----------|------|------|
| 当前方案（8 分频） | 12.5 MHz | clk_cnt[2] 8 分频 | — | 无（已验证） |
| 减少分频比（4 分频） | 25 MHz | 改为 clk_cnt[1] 4 分频 | 低 | 需重新验证时序（33.753ns < 40ns，满足） |
| 减少分频比（2 分频） | 50 MHz | 改为 clk_cnt[0] 2 分频 | 低 | 需重新验证时序（33.753ns > 20ns，可能违例） |
| 使用 PLL/MMCM | 50-100 MHz | 用 MMCM 生成高频时钟 | 中 | 需验证关键路径是否满足 |
| 优化关键路径 | 30+ MHz | 流水化 ALU、拆分组合逻辑 | 高 | 需修改 RTL 并重新验证 |

> **注意**：当前 12.5 MHz 方案已在 Nexys4 DDR 板上通过全部 8 项测试验证，建议保持当前配置。若需提升频率，需重新进行时序分析和板级验证。

### 4.3 false_path 约束建议

当前 XDC 已添加以下 false_path 约束，确保时序分析准确性：

| 约束 | 路径 | 原因 |
|------|------|------|
| 跨时钟域 false_path | clk_soc ↔ sys_clk_pin | pc_dbg 等调试信号为慢变信号，无需同步时序 |
| 七段数码管 false_path | → seg_ca~cg, an[*] | 慢速显示设备，1 kHz 刷新率，无需纳秒级时序 |
| 异步复位 false_path | rst_n → all_registers | 异步复位信号，不满足同步时序要求 |

> **建议**：未来若增加新的跨时钟域信号（如中断、DMA 请求等），需同步添加对应的 false_path 约束或使用同步器电路，避免 Vivado 产生虚假时序违例。
