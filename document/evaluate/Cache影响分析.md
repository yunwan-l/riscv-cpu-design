# Cache 影响分析

## 1. 当前存储系统架构

### 1.1 存储器配置

RVP SoC 当前采用 CPU → I-Cache → 后备 BRAM 的三级取指路径，数据访问仍为直接寻址片上存储器：

| 存储器 | 容量 | 访问方式 | 延迟 | 实现方式 |
|--------|------|----------|------|----------|
| I-Cache（指令缓存） | 256 字节（64 行 x 1 字） | 异步读、同步填充 | 1 周期（命中/缺失均同周期返回） | 分布式 RAM / 寄存器阵列 |
| 指令存储器（IMEM，后备） | 8 KB（2048 字） | 异步读 | 1 周期 | BRAM |
| 数据存储器（DMEM） | 8 KB（2048 字） | 异步读、同步写 | 1 周期 | BRAM |

取指路径：CPU 取指地址（PC）首先送入 I-Cache（`rvp_icache.sv`），命中时直接从缓存寄存器阵列返回指令；未命中时从后备指令存储器（`rvp_instr_mem.sv`）异步读取，返回该指令同时在时钟上升沿同步填充对应缓存行。

### 1.2 访问特性

当前存储系统的关键特性：

- **异步读（I-Cache 命中）**：PC 送入 I-Cache 后，Tag 比较为组合逻辑，命中时数据在当周期内从缓存寄存器阵列立即可用
- **异步读（I-Cache 缺失）**：未命中时从后备 BRAM 异步读取，数据同样在当周期内返回，同时在时钟上升沿同步填充对应缓存行
- **零缺失惩罚**：由于后备 BRAM 同为异步读，命中和缺失均在同一周期返回数据，I-Cache 不引入任何流水线停顿
- **数据存储器单周期访问**：Load 指令在 MEM 阶段读取数据，数据在当周期即可前递到 EX 阶段的下一条指令（需 1 周期 Load-Use 停顿）
- **缓存统计**：I-Cache 内部维护命中/缺失计数器（`hit_count` / `miss_count`），可通过性能计数器总线读取（地址 `0x10030014` / `0x10030018`）

### 1.3 等效模型

当前存储系统取指路径可等效为：

```
I-Cache 模型（已实现）：
  容量 = 256 字节（64 行 x 1 字）
  关联度 = 直接映射
  命中延迟 = 1 周期（缓存寄存器组合读）
  缺失延迟 = 1 周期（后备 BRAM 组合读，同周期返回）
  缺失惩罚 = 0 周期（不产生 stall）
  平均访问时间 = 1 周期（与命中率无关）
  冷启动缺失 = 存在（首次访问时 Valid=0，但无额外周期代价）
```

## 2. I-Cache 实际实现分析

### 2.1 I-Cache 架构参数（已实现）

I-Cache 已在 `rvp_icache.sv` 中实现，以下为实际架构参数：

| 参数 | 值 | 说明 |
|------|-----|------|
| Cache 容量 | 256 字节 | 64 行 x 1 字 |
| 块大小 | 4 字节（1 条指令） | 每行缓存 1 条指令 |
| 行数 | 64 | 索引位宽 6 位 |
| 关联度 | 直接映射 | 简单硬件实现 |
| 地址分解 | Tag[31:8] \| Index[7:2] \| Offset[1:0]=00 | — |
| 命中延迟 | 1 周期 | 缓存寄存器组合读 |
| 缺失延迟 | 1 周期 | 后备 BRAM 组合读，同周期返回 |
| 缺失惩罚 | 0 周期 | 不产生流水线停顿 |
| 填充方式 | 同步写（时钟上升沿） | 未命中时在当周期上升沿填充 |
| 后备存储 | rvp_instr_mem（2048 字 = 8KB BRAM） | 异步读 |
| 统计计数器 | hit_count / miss_count | 可通过性能计数器总线读取 |

> 注：此前的理论分析假设 I-Cache 容量为 4KB、块大小 32 字节、缺失惩罚 8 周期。实际实现的参数远小于理论假设，且由于后备 BRAM 也为异步读，缺失惩罚为零。详见第 7 节对比。

## 3. 加入 D-Cache（数据 Cache）的影响分析

> 注：D-Cache 目前尚未实现，以下为理论假设分析。

### 3.1 D-Cache 架构假设

| 参数 | 值 | 说明 |
|------|-----|------|
| Cache 容量 | 4 KB | 典型 L1 D-Cache 大小 |
| 块大小 | 32 字节（8 个字） | 每次缺失加载 8 个字 |
| 关联度 | 2 路组相联 | 平衡命中率和硬件复杂度 |
| 写策略 | 写直达（Write-Through） | 简化一致性管理 |
| 命中延迟 | 1 周期 | 与当前存储器相同 |
| 缺失惩罚 | 8 周期 | 从后备存储加载 8 个字的延迟 |

### 3.2 基准测试程序的 D-Cache 访问模式分析

#### 矩阵乘法

| 访问模式 | 分析 |
|----------|------|
| 数据总量 | A(36B) + B(36B) + C(36B) = 108 字节 < 4 KB |
| 访问序列 | 内层循环：lw A[i][k], lw B[k][j], sw C[i][j] |
| 空间局部性 | 矩阵按行存储，内层 k 递增时 A[i][k] 连续访问 |
| 时间局部性 | A 和 B 在不同 i,j 迭代中被重复访问 |
| Load 指令数 | 27 次 lw A + 27 次 lw B = 54 次 |
| 预估命中率 | > 98%（数据量远小于 Cache） |
| 缺失次数 | 首次加载 A(5 行) + B(5 行) + C(5 行) ≈ 15 次 |
| D-Cache 对 CPI 影响 | +0.178（15 次缺失 × 8 周期 / 674 指令） |

#### 冒泡排序

| 访问模式 | 分析 |
|----------|------|
| 数据总量 | 8 个元素 = 32 字节 < 4 KB |
| 访问序列 | 内层循环：lw arr[j], lw arr[j+1], sw arr[j], sw arr[j+1] |
| 空间局部性 | 连续访问相邻元素 arr[j] 和 arr[j+1] |
| 时间局部性 | 同一元素在多次内层迭代中被访问 |
| Load 指令数 | 每次内层迭代 2 次 lw，共约 56 次 |
| 预估命中率 | > 99%（数据量仅 32 字节 = 1 个 Cache 行） |
| 缺失次数 | 首次加载数组 ≈ 1 次 |
| D-Cache 对 CPI 影响 | +0.017（1 次缺失 × 8 周期 / 482 指令） |

| 访问模式 | 分析 |

| 空间局部性 | 顺序写入连续地址 |
| Load 指令数 | 0 次 lw（纯写程序） |

### 3.3 D-Cache 影响总结

| 程序 | 当前 CPI | 加 D-Cache 后 CPI | CPI 变化 | 说明 |
|------|----------|-------------------|----------|------|
| 矩阵乘法 | 1.0846 | 1.263 | +0.178 (+16.4%) | 冷启动缺失较多 |

**结论**：对于小规模数据集，D-Cache 的主要开销来自冷启动缺失（首次加载数据时的强制缺失）。矩阵乘法受影响最大（+16.4% CPI），因为其数据量最大且访问模式更复杂。冒泡排序和斐波那契几乎不受影响。

## 4. 在现有 I-Cache 基础上加入 D-Cache 的综合影响

### 4.1 综合 CPI 影响

I-Cache 已实现且对 CPI 贡献为零，因此综合影响完全来自 D-Cache：

| 程序 | 当前 CPI（含 I-Cache） | 加 D-Cache 后 CPI | CPI 变化 | I-Cache 贡献 | D-Cache 贡献 |
|------|----------|----------------|----------|-------------|-------------|
| 矩阵乘法 | 1.0846 | 1.263 | +0.178 (+16.4%) | 0.000 | +0.178 |
| 冒泡排序 | 1.1826 | 1.199 | +0.017 (+1.4%) | 0.000 | +0.017 |
| 斐波那契 | 1.1529 | 1.153 | +0.000 (+0.0%) | 0.000 | +0.000 |

### 4.2 分析结论

对于当前的小规模基准测试程序，加入 D-Cache 会**增加** CPI 而非降低，原因如下：

1. **当前 I-Cache 零缺失惩罚**：I-Cache 和后备 BRAM 均为异步读，等效于 100% 命中率的完美 Cache，不引入任何停顿周期
2. **数据存储器已是单周期访问**：数据 BRAM 同为异步读，等效于 100% 命中率的完美 D-Cache，任何实际 D-Cache 的命中率都低于 100%
3. **程序和数据量极小**：所有基准测试程序（最大 304 字节代码 + 108 字节数据）远小于 4 KB Cache 容量，但 D-Cache 的冷启动缺失仍会引入额外周期
4. **缺失惩罚不可忽略**：即使缺失率很低（< 2%），每次缺失 8 周期的惩罚对 CPI 的影响仍超过 10%（以矩阵乘法为例）

### 4.3 Cache 有益的场景

Cache 在以下场景中才能带来正面收益：

| 场景 | 当前存储器 | Cache 方案 | Cache 优势 |
|------|-----------|-----------|-----------|
| 大程序（>8KB） | 无法容纳 | I-Cache + 外部 Flash | 命中时 1 周期 vs 外部 10+ 周期 |
| 大数据集（>8KB） | 无法容纳 | D-Cache + 外部 SRAM | 命中时 1 周期 vs 外部 5+ 周期 |
| 外部 DRAM 后备 | 50-100 周期 | L1 Cache + DRAM | 命中时 1 周期 vs DRAM 50+ 周期 |
| 多任务切换 | 每次全加载 | Cache + 替换策略 | 热数据保留在 Cache 中 |

## 5. 不同存储层级方案对比

### 5.1 方案对比

| 方案 | 命中延迟 | 缺失延迟 | 平均 CPI 贡献 | 适用场景 |
|------|----------|----------|-------------|----------|
| 当前（I-Cache + BRAM 后备，零缺失惩罚） | 1 周期 | 1 周期 | 1.000 | 小程序，数据量 < 8KB |
| 当前 + D-Cache（BRAM 后备） | 1 周期 | 2-3 周期 | 1.001-1.010 | 中等数据量，偶尔缺失 |
| L1 Cache + 外部 SRAM | 1 周期 | 5-10 周期 | 1.050-1.150 | 大程序，频繁缺失 |
| L1 Cache + 外部 DRAM | 1 周期 | 50-100 周期 | 1.500-2.000 | 超大程序，高缺失率 |
| L1 + L2 Cache | 1 周期 | L2: 5-10 周期 | 1.010-1.030 | 大程序，低缺失率 |

### 5.2 推荐方案

对于当前 RVP SoC 的典型应用场景（嵌入式小规模程序）：

- **当前方案（I-Cache + BRAM 后备）已最优**：I-Cache 零缺失惩罚，程序和数据量均小于片上存储容量，1 周期访问无缺失
- **不建议添加 D-Cache**：数据存储器使用 BRAM 直接实现已足够，D-Cache 的写策略复杂性和冷启动缺失惩罚不值得为小数据集引入
- **未来扩展**：若程序规模增长超过 8KB，建议扩展 I-Cache 容量或使用外部存储器作为后备，利用现有 I-Cache 框架（填充逻辑已就绪）

## 6. 总结

### 6.1 关键发现

1. **I-Cache 已实现且零缺失惩罚**：命中和缺失均从异步读路径返回数据（1 周期），对 CPI 影响为零
2. **加入 D-Cache 会增加 CPI 0%~16.4%**：冷启动缺失的额外周期无法被低缺失率抵消（假设缺失惩罚 8 周期）
3. **I-Cache 对短程序无负面影响**：与理论分析不同，实际实现的 I-Cache 缺失惩罚为零，斐波那契等短程序不受影响
4. **D-Cache 对大数据量影响大**：矩阵乘法 108 字节数据，15 次冷启动缺失导致 +16.4% CPI
5. **Cache 的价值在规模化场景**：当程序或数据超过片上存储容量时，Cache 的平均访问时间优势才能体现

### 6.2 对 CPI 的量化影响

| 方案 | 矩阵乘法 CPI | 冒泡排序 CPI | 斐波那契 CPI | 平均 CPI |
|------|-------------|-------------|-------------|----------|
| 当前（I-Cache 实现） | 1.0846 | 1.1826 | 1.1529 | 1.1400 |
| +D-Cache（理论假设） | 1.263 (+16.4%) | 1.199 (+1.4%) | 1.153 (+0.0%) | 1.205 (+5.7%) |

> 注：I-Cache 已实现，对 CPI 贡献为 0.000（零缺失惩罚）。D-Cache 影响为理论估算值，基于冷启动强制缺失（compulsory miss）模型，假设缺失惩罚 8 周期，未考虑容量缺失和冲突缺失。实际 D-Cache 命中率可能因 Cache 组织方式和替换策略而有所不同。

### 7.2 CPI 影响对比

| 程序 | 理论估算 CPI 变化 | 实际 CPI 变化 | 差异原因 |

| 矩阵乘法 | +0.015 (+1.4%) | 0.000 (0%) | 实际缺失惩罚为零 |

理论分析假设缺失惩罚为 8 周期（从后备存储加载 32 字节），因此即使少量缺失也会显著影响 CPI。实际实现中，后备 BRAM 同为异步读，命中和缺失均在同一周期返回数据，I-Cache 对 CPI 的影响完全为零。

### 7.3 I-Cache 的实际价值

尽管 I-Cache 对 CPI 无影响，但其实现仍有实际意义：

1. **减少 BRAM 读端口竞争**：命中时从缓存寄存器阵列读取，不访问后备 BRAM，降低 BRAM 端口压力
2. **缓存统计功能**：内置命中/缺失计数器，可通过性能计数器总线（`0x10030014` / `0x10030018`）读取，用于命中率分析和程序行为 profiling
3. **架构可扩展性**：填充逻辑已就绪，未来若将后备存储改为外部 SRAM/Flash（多周期访问），I-Cache 将自动发挥缺失惩罚屏蔽作用
4. **设计验证完整**：I-Cache 作为完整存储层级的一部分，验证了 Cache 设计流程的可行性

### 7.4 板级验证结果

在 Nexys4 DDR 开发板上完成全系统验证，I-Cache 工作正常：

- **测试 3（前递链测试）**：通过缓存代码路径正确执行，验证 I-Cache 命中/缺失路径的数据正确性
- **全部 8 项测试通过**：涵盖算术运算、存储访问、分支跳转、前递链等功能，I-Cache 对所有测试透明
- **性能计数器验证**：I-Cache 命中/缺失计数器可通过总线正确读取，统计数据合理

### 7.5 结论

实际 I-Cache 实现与理论分析的核心差异在于**缺失惩罚**：理论假设 8 周期，实际为 0 周期。这一差异的根本原因是后备 BRAM 采用异步读（组合逻辑读），与 I-Cache 缓存阵列的读取方式一致，因此 I-Cache 的引入不改变任何时序行为，对流水线完全透明。I-Cache 的主要价值在于架构完整性、统计功能和未来可扩展性，而非当前的 CPI 优化。
