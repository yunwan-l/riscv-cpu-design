# 瓶颈分析与优化方案

## 1. 性能瓶颈识别

### 1.1 瓶颈排序

基于 CPI 分解分析，三个基准测试程序的性能损失来源按影响程度排序：

| 排名 | 瓶颈类型 | 平均 CPI 贡献 | 占额外 CPI 比例 | 影响程序 |
|------|----------|-------------|----------------|----------|
| 1 | 分支冲刷（控制冒险） | 0.0920 | 66.2% | 全部 3 个程序 |
| 2 | Load-Use 停顿（数据冒险） | 0.0327 | 23.5% | 矩阵乘法、冒泡排序 |
| 3 | 流水线排空开销 | 0.0153 | 11.0% | 全部 3 个程序（短程序影响更大） |

### 1.2 分支冲刷瓶颈详解

**根本原因**：分支指令在 EX 阶段（第 3 级）才判定是否跳转，此时 IF 和 ID 阶段已各取/译码了一条指令。冒险单元仅冲刷 IF/ID 寄存器（1 条指令），但被冲刷的指令浪费了 1 个时钟周期。

**冲刷机制**：

```
时钟周期:    1     2     3     4     5     6     7
分支指令:   [IF]  [ID]  [EX]  [MEM] [WB]
下一条:           [IF]  [ID]  ← 冲刷（flush_if_id=1）
正确目标:               [IF]  [ID]  [EX]  [MEM] [WB]

损失: 1 个时钟周期（周期 3 的 IF 被冲刷）
```

**冲刷惩罚量化**：

| 程序 | 分支指令数 | 跳转次数 | 每次跳转惩罚 | 总冲刷周期 | 占额外 CPI |
|------|-----------|----------|-------------|-----------|-----------|
| 矩阵乘法 | 40 | 27 | 1 周期 | 27 | 47.4% |
| 冒泡排序 | 99 | 57 | 1 周期 | 57 | 64.8% |
| 斐波那契 | 11 | 10 | 1 周期 | 10 | 76.9% |

### 1.3 Load-Use 停顿瓶颈详解

**根本原因**：Load 指令在 MEM 阶段（第 4 级）才能读到数据，若下一条指令在 EX 阶段（第 3 级）需要该数据，前递无法跨越 MEM→EX 的时序边界，需要插入 1 个停顿周期。

**停顿机制**：

```
时钟周期:    1     2     3     4     5     6     7
lw 指令:    [IF]  [ID]  [EX]  [MEM] [WB]
                              ↑ 数据可用
依赖指令:          [IF]  [ID]  [停顿] [EX]  [MEM] [WB]
                               ↑ stall=1, 等待前递

损失: 1 个时钟周期
```

**停顿惩罚量化**：

| 程序 | 停顿次数 | 每次 停顿惩罚 | 总停顿周期 | 占额外 CPI |
|------|----------|-------------|-----------|-----------|
| 矩阵乘法 | 27 | 1 周期 | 27 | 47.4% |
| 冒泡排序 | 28 | 1 周期 | 28 | 31.8% |
| 斐波那契 | 0 | 1 周期 | 0 | 0.0% |

**停顿来源分析**：

- 矩阵乘法：`lw t0, 0(t5)` → `mul t4, t0, t1`（内层循环，27 次迭代）
- 冒泡排序：`lw t0, 0(t4)` → `ble t0, t1`（内层循环比较，28 次触发）

### 1.4 流水线排空开销详解

**根本原因**：程序结束时执行 `j end` 死循环，停机检测在 3 周期后触发。此时流水线中仍有指令未完成排空，导致少量额外周期。

**开销量化**：

| 程序 | 总周期 | 指令数 | 停顿+冲刷 | 排空开销 | 占额外 CPI |
|------|--------|--------|----------|----------|-----------|
| 矩阵乘法 | 731 | 674 | 54 | 3 | 5.2% |
| 冒泡排序 | 570 | 482 | 85 | 3 | 3.4% |
| 斐波那契 | 98 | 85 | 10 | 3 | 23.1% |

> 排空开销固定为 3 周期，对短程序（斐波那契 85 条指令）影响较大（23.1%），对长程序影响微小。

## 2. 优化方案设计

### 2.1 方案一：分支判定提前至 ID 阶段

**当前状态**：分支在 EX 阶段判定，跳转惩罚 1 周期（冲刷 IF/ID 中的 1 条指令）

**优化目标**：分支在 ID 阶段判定，跳转惩罚降为 0 周期（无需冲刷）

**实现方式**：

1. 在 ID 阶段添加分支比较器，提前计算分支条件
2. 修改冒险单元：分支跳转时无需冲刷 IF/ID（因为下一条指令尚未进入 ID）
3. 需要前递逻辑支持 ID 阶段读取操作数（从 EX/MEM/WB 前递到 ID）

**架构变更**：

```
当前:  IF → ID(译码) → EX(分支判定) → MEM → WB
优化后: IF → ID(译码+分支判定) → EX → MEM → WB
```

**预期收益**：

| 程序 | 当前冲刷 | 优化后冲刷 | 减少周期 | CPI 降低 | 新 CPI |
|------|----------|-----------|----------|----------|--------|
| 矩阵乘法 | 27 | 0 | 27 | 0.0401 | 1.0445 |
| 冒泡排序 | 57 | 0 | 57 | 0.1183 | 1.0643 |
| 斐波那契 | 10 | 0 | 10 | 0.1176 | 1.0353 |

**实现复杂度**：中

**风险**：
- ID 阶段增加比较器逻辑可能延长关键路径，降低 Fmax
- 需要新增 ID 阶段的前递路径，增加数据通路复杂度
- 需要处理 ID 阶段分支判定与 Load-Use 冒突的情况（若分支依赖 Load 结果，仍需停顿）

### 2.2 方案二：添加 2-bit 饱和分支预测器

**当前状态**：无分支预测，所有分支跳转均产生 1 周期冲刷

**优化目标**：预测分支行为，减少错误预测导致的冲刷

**实现方式**：

1. 添加分支历史表（BHT），每个分支地址对应一个 2-bit 饱和计数器
2. IF 阶段根据 PC 查询 BHT，预测是否跳转
3. EX 阶段验证预测，若预测错误则冲刷（惩罚 1 周期），若正确则无惩罚
4. BHT 大小：64 项（6 位 PC 索引），每项 2 bit，总计 128 bit

**2-bit 饱和预测器状态机**：

```
    强不跳转 (00) ←──── 不跳转 ──── 弱不跳转 (01)
         │                              │
       跳转                          跳转
         │                              │
         ↓                              ↓
    强跳转 (11) ←──── 跳转 ──── 弱跳转 (10)
         │                              │
       不跳转                        不跳转
         │                              │
         └──────────────────────────────┘
```

**预期收益**：

基于三个程序的分支跳转模式分析：

| 程序 | 分支类型 | 跳转率 | 预测准确率（估） | 优化后冲刷 | 减少率 |
|------|----------|--------|-----------------|-----------|--------|
| 矩阵乘法 | 循环 bne | 67.5% | ~89%（循环模式规律） | ~3 | 88.9% |
| 冒泡排序 | 循环 bge/ble/j/bne | 57.6% | ~75%（分支模式复杂） | ~14 | 75.4% |
| 斐波那契 | 循环 bne | 90.9% | ~91%（几乎总跳转） | ~1 | 90.0% |

**预估新 CPI**：

| 程序 | 当前 CPI | 减少冲刷 | 新 CPI | 改善幅度 |
|------|----------|----------|--------|----------|
| 矩阵乘法 | 1.0846 | -0.036 | 1.049 | 3.3% |
| 冒泡排序 | 1.1826 | -0.089 | 1.094 | 7.5% |
| 斐波那契 | 1.1529 | -0.106 | 1.047 | 9.2% |

**实现复杂度**：中高

**风险**：
- 需要 IF 阶段增加 BHT 查询逻辑
- 需要处理预测错误时的流水线冲刷（仍需 1 周期惩罚）
- BHT 初始化和冷启动（cold start）期间的预测准确率较低

### 2.3 方案三：提高运行频率

**当前状态**：CPU 核心运行在 50 MHz（clk_div2），理论 Fmax = 228.5 MHz

**优化目标**：提高运行频率至 100 MHz 或更高

**实现方式**：

| 子方案 | 目标频率 | 实现方式 | 复杂度 |
|--------|----------|----------|--------|
| 3a. 直通时钟 | 100 MHz | 移除 clk_div2 分频器，CPU 直接使用 100 MHz | 低 |
| 3b. PLL 倍频 | 200 MHz | 使用 MMCM 生成 200 MHz 时钟 | 低 |
| 3c. 修复外设时序 | 100 MHz | 将七段数码管迁移至慢时钟域 + 方案 3a | 中 |

**预期收益**：

| 频率 | 矩阵乘法 MIPS | 冒泡排序 MIPS | 斐波那契 MIPS | 平均 MIPS |
|------|-------------|-------------|-------------|-----------|
| 50 MHz（当前） | 46.10 | 42.28 | 43.37 | 43.92 |
| 100 MHz | 92.20 | 84.56 | 86.74 | 87.84 |
| 200 MHz | 184.40 | 169.12 | 173.48 | 175.67 |
| 228.5 MHz（理论） | 210.69 | 193.22 | 198.20 | 200.44 |

**实现复杂度**：低（方案 3a）至中（方案 3c）

**风险**：
- 方案 3a 需验证 CPU 在 100 MHz 下时序是否满足（当前关键路径 4.377 ns < 10 ns，应该满足）
- 方案 3a 需修复 sys_clk_pin 域的七段数码管时序违例
- 方案 3b 需验证 200 MHz 下时序（4.377 ns < 5 ns，满足）

### 2.4 方案四：Load-Use 调度优化（已实施）

**当前状态**：Load 后紧跟依赖指令时插入 1 周期停顿

**优化目标**：通过编译器调度减少 Load-Use 停顿

**实现方式**：

1. 汇编器/编译器在 Load 和依赖指令之间插入独立指令
2. 利用循环不变量外提、指令重排等优化

**示例（矩阵乘法内层循环）**：

```assembly
# 优化前（产生 Load-Use 停顿）：
lw t1, 0(t5)       # Load B[k][j]
mul t4, t0, t1     # 依赖 t1 → 停顿 1 周期
add t2, t2, t4
addi a2, a2, 1     # k++

# 优化后（重排指令消除停顿）：
lw t1, 0(t5)       # Load B[k][j]
addi a2, a2, 1     # 独立指令（k++），填充停顿槽
mul t4, t0, t1     # t1 已就绪（MEM/WB 前递），无停顿
add t2, t2, t4
```

**实际收益**（仿真验证，6/6 PASS）：

| 程序 | 原版停顿 | 优化后停顿 | 原版 CPI | 优化后 CPI | CPI 改善 |
|------|----------|-----------|----------|-----------|----------|
| 矩阵乘法 | 27 | 0 | 1.0846 | 1.0445 | -3.70% |
| 冒泡排序 | 28 | 0 | 1.1826 | 1.1207 | -5.24% |
| 斐波那契 | 0 | 0 | 1.1529 | 1.1529 | 0.00% |

> Load-Use 停顿完全消除（27/28 → 0），矩阵乘法周期数减少 27（731→704），冒泡排序周期数减少 13（570→557）。平均 MIPS 从 43.92 提升至 45.28（+3.10%）。

**实现复杂度**：高（需改进汇编器添加指令调度 Pass）

**风险**：
- 指令调度可能改变程序语义，需要仔细验证 — 已通过 6/6 数据验证确认无误
- 部分场景无法找到合适的独立指令填充停顿周期 — 斐波那契无 Load-Use 停顿，无法优化

## 4. 方案一详细设计（分支判定提前至 ID）

### 4.1 架构变更

**当前 ID 阶段**：指令译码 + 寄存器读取

**优化后 ID 阶段**：指令译码 + 寄存器读取 + 分支条件计算

需要在 ID 阶段增加：
1. 分支比较器（比较 rs1 和 rs2 的值）
2. ID 阶段前递逻辑（从 EX/MEM、MEM/WB 前递到 ID）
3. 修改冒险单元：分支跳转时冲刷 IF（而非 IF/ID），无需冲刷 ID 中的指令

### 4.2 数据通路变更

```
                    ┌─────────────────────────────────────────┐
                    │              前递逻辑                     │
                    │  EX/MEM → ID, MEM/WB → ID                │
                    └────┬──────────────┬──────────────────────┘
                         │              │
                    ┌────▼──────────────▼──────┐
 IF阶段 ──────────► │  ID阶段                   │ ──► EX阶段
 (取指)             │  - 译码                   │     (ALU运算)
                    │  - 寄存器读取              │
                    │  - 分支条件计算（新增）     │
                    │  - 分支目标地址计算（已有） │
                    └───────────────────────────┘
```

### 4.3 冒险处理变更

| 冒险类型 | 当前处理 | 优化后处理 |
|----------|----------|-----------|
| 分支跳转 | EX 判定，冲刷 IF/ID（1 周期） | ID 判定，冲刷 IF（0 周期） |
| Load-Use | 停顿 1 周期 | 不变 |
| 分支依赖 Load | 无特殊处理 | 需额外停顿 1 周期（等待 Load 数据） |

### 4.4 关键代码变更（伪代码）

```systemverilog
// rvp_hazard_unit.sv 变更

// 当前：
assign flush_if_id = (branch_taken || take_jump) && !stall;

// 优化后：
// 分支在 ID 判定，跳转时冲刷 IF（PC 重定向）
assign flush_if_id = (id_branch_taken || id_take_jump) && !stall;
// 不再需要 flush_id_ex 用于分支（仅用于 stall）
assign flush_id_ex = stall;

// 新增：ID 阶段分支条件计算
// 在 rvp_core_pipeline.sv 的 ID 阶段添加：
logic id_branch_taken;
always_comb begin
  if (is_branch_id) begin
    case (funct3_id)
      BEQ:  id_branch_taken = (rs1_data_id == rs2_data_id);
      BNE:  id_branch_taken = (rs1_data_id != rs2_data_id);
      BLT:  id_branch_taken = ($signed(rs1_data_id) < $signed(rs2_data_id));
      BGE:  id_branch_taken = ($signed(rs1_data_id) >= $signed(rs2_data_id));
      // ...
    endcase
  end
end
```

| 矩阵乘法 CPI | 1.0846 | 1.0445 | -3.7% |

## 5. 总结

### 5.1 瓶颈优先级

```
分支冲刷 (66.2%) >>>>>> Load-Use 停顿 (23.5%) >> 流水线开销 (11.0%)
```

### 5.2 优化方案优先级

```
提频至100MHz (2×吞吐, 低复杂度) >> 分支提前至ID (8.1% CPI降低, 中复杂度) 
> 分支预测器 (7% CPI降低, 中高复杂度) > 指令调度 (3% CPI降低, 高复杂度)
```

### 5.3 最终建议

**最优组合**：方案 3a（提频至 100 MHz）+ 方案 1（分支提前至 ID）

- 预期平均 CPI：1.048
- 预期运行频率：100 MHz
- 预期平均吞吐量：95.42 MIPS
- 相比当前（43.92 MIPS）提升：2.17×

此组合以较低的实现复杂度获得最大的性能提升，是阶段三优化的推荐方向。

### 6.6 UART 串口验证

- 采用 **UART 串口（115200 baud）** 替代 LED 观察方式
- CPU 运算结果通过 UART 精确回传至上位机，可精确验证运算结果的正确性
- 相比 LED 观察，UART 方式支持复杂数据（如矩阵元素、排序数组）的完整输出与校验
