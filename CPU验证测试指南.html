RVP处理器 FPGA板级验证测试指南

NEXYS4 DDR | Artix-7 XC7A100T | 25MHz | RV32IM 5级流水线

使用方法：运行 switch_firmware.bat，选择测试编号，脚本自动将对应固件复制为 firmware.hex。然后在 Vivado 中依次点击 Run Synthesis → Run Implementation → Generate Bitstream，最后通过 Hardware Manager 烧录到板子。
重要：每次切换固件必须重新综合，因为固件数据在综合阶段通过 $readmemh 烧入 BRAM。仅重新生成比特流是不够的。

测试2 PC顺序增长

固件

firmware_pc_seq.hex — 8条指令

验证内容

验证PC按PC+4正确递增。程序循环执行 addi x6, x6, 1（计数器+1），将计数器值输出到LED。如果PC递增逻辑错误，计数器不会正常递增。

程序流程

0x000: lui  x5, 0x10010    # GPIO基地址
0x004: addi x6, x0, 0      # 计数器=0
0x008: addi x6, x6, 1      # 计数器++   ← 循环体
0x00C: sw   x6, 0(x5)      # LED = 计数器
0x010: lui  x10, 0x100     # 延迟循环开始
0x014: addi x10, x10, -1   # x10--
0x018: bne  x10, x0, -4    # 延迟循环
0x01C: jal  x0, -20        # 跳回0x008

预期LED表现

LED显示二进制计数递增：0x0001 → 0x0002 → 0x0003 → ... → 0x000F → 0x0010 → ...，每个值保持约160ms，频率约6Hz。计数器持续递增，证明PC在循环体内正确递增。

0x0001 (L0亮)

然后变化为：

0x0002 (L1亮)

持续递增...直到所有LED滚动变化。

测试3 流水线前递验证

固件

firmware_forward.hex — 19条指令

验证内容

8条连续数据依赖的指令链，每条指令使用上一条的结果。在5级流水线中，第N+1条指令在EX级需要第N条指令的结果，而第N条指令此时在MEM级，必须通过MEM→EX前递通路获取数据。

程序流程

addi x1, x0, 1     # x1 = 1
addi x2, x1, 1     # x2 = 2  (前递x1: MEM→EX)
addi x3, x2, 1     # x3 = 3  (前递x2: MEM→EX)
addi x4, x3, 1     # x4 = 4  (前递x3: MEM→EX)
addi x6, x4, 1     # x6 = 5  (前递x4: MEM→EX)
addi x7, x6, 1     # x7 = 6  (前递x6: MEM→EX)
addi x8, x7, 1     # x8 = 7  (前递x7: MEM→EX)
addi x9, x8, 1     # x9 = 8  (前递x8: MEM→EX)
slli x10, x9, 8   # x10 = 8<<8 = 0x0800
sw   x10, 0(x5)   # LED = 0x0800

预期LED表现

通过：LED11闪烁（0x0800 ↔ 0x0000，约3Hz），表示前递链全部正确，x9=8。

0x0800 (L11亮) → 闪烁

失败：如果前递通路断开，某条addi读到旧值（0），后续全部错误，LED显示的值将不是0x0800。

测试4 Load-Use冒险验证

固件

firmware_loaduse.hex — 14条指令

验证内容

测试Load-Use冒险检测和流水线停顿。程序向RAM地址0写入0xAA，然后lw读取，立即在下一步使用（addi x3, x2, 0）。由于lw的数据在MEM级才返回，而后一条指令在EX级就需要，必须停顿1拍。

程序流程

addi x1, x0, 0xAA   # x1 = 0xAA
sw   x1, 0(x0)       # RAM[0] = 0xAA
lw   x2, 0(x0)       # x2 = RAM[0] (load, 数据MEM级才出)
addi x3, x2, 0       # x3 = x2  (Load-Use! 需停顿1拍)
sw   x3, 0(x5)       # LED = x3

预期LED表现

通过：LED交替显示 0x00AA 和 0x0000（约3Hz闪烁），表示冒险停顿正确，x3=0xAA。

0x00AA (L1,L2,L3,L5,L6,L7亮) → 闪烁

失败：如果冒险检测失效，addi读到的是x2的旧值（0），LED显示0x0000（全灭不闪）或错误值。

测试5 分支跳转验证

固件

firmware_branch.hex — 21条指令

验证内容

测试3种条件分支的正确性：BNE（不等于跳转）、BEQ（等于跳转，应不跳转）、BLT（小于跳转）。如果分支逻辑正确，错误路径代码被跳过，LED交替显示两种成功模式。

程序流程

addi x1, x0, 5     # x1 = 5
addi x2, x0, 3     # x2 = 3

bne  x1, x2, +12   # 5≠3 → 跳转(应跳)  → 跳过错误代码
addi x6, x0, 0xFF  # (不应执行)
sw   x6, 0(x5)     # (不应执行)

beq  x1, x2, +12   # 5≠3 → 不跳(应不跳) → 继续
blt  x2, x1, +12   # 3<5 → 跳转(应跳)   → 跳过错误代码
addi x6, x0, 0xF0  # (不应执行)
sw   x6, 0(x5)     # (不应执行)

addi x6, x0, 0x0F  # 成功! x6 = 0x000F
sw   x6, 0(x5)     # LED = 0x000F

预期LED表现

通过：LED在 0x000F（低4位亮）和 0x00F0（高4位亮）之间交替，约3Hz。

0x000F (L0-L3亮)

然后切换为：

0x00F0 (L4-L7亮)

失败：如果分支方向错误，错误代码会执行，LED可能出现0x00FF或异常模式。

测试9 流水线综合演示

固件

firmware_pipeline.hex — 29条指令

验证内容

综合演示流水线4个核心机制，每个阶段用不同方式计算同一个递增值，LED依次显示1→2→3→4循环。

4个阶段说明

阶段LED值验证的流水线机制关键指令
10x0001基础指令（无依赖）addi x1, x0, 1 → sw
20x0002数据前递（MEM→EX）addi x2,x0,1 → addi x2,x2,1
30x0003Load-Use冒险停顿sw x3 → lw x4 → addi x4,x4,0
40x0004分支冲刷（BNE taken）bne x1,x2,+8 → 跳过addi

预期LED表现

LED循环显示 0x0001 → 0x0002 → 0x0003 → 0x0004 → 0x0001 → ...，每个值约160ms。4个值依次出现，证明流水线的前递、冒险停顿、分支冲刷全部正常工作。

0x0001 → 0x0002 → 0x0003 → 0x0004 → 循环

操作步骤

首次使用

  1. 确保 Vivado 项目已通过 create_project.tcl 创建完成
  2. 将 switch_firmware.bat 和所有 firmware_*.hex 文件放在同一目录（即 rvp_build 目录）
  3. 双击运行 switch_firmware.bat
  4. 选择要测试的固件编号（1-9）
  5. 脚本自动将选中的固件复制为 firmware.hex
  6. 回到 Vivado，点击 Run Synthesis（等待综合完成）
  7. 点击 Run Implementation（等待实现完成）
  8. 点击 Generate Bitstream（等待比特流生成完成）
  9. 打开 Hardware Manager，连接 NEXYS4 DDR，烧录比特流
  10. 观察 LED 和数码管表现，与本文档的预期对比

切换测试

  1. 重新运行 switch_firmware.bat，选择新的固件编号
  2. 在 Vivado 中重新执行 Synthesis → Implementation → Bitstream
  3. 烧录新的比特流，观察表现
常见问题：如果烧录后 LED 全灭或无反应，请检查：
1. firmware.hex 是否已更新（检查文件修改时间）
2. 是否重新执行了 Synthesis（不能只重新生成比特流）
3. 综合日志中是否有 $readmemh 成功加载的提示
